```md
# NON_NEGOTIABLES


## Purpose
This document defines the hard non-negotiables for any design-system or UX-system work on CatchIt.
These are not suggestions. They are constraints.


---


## 1. Product identity
CatchIt is **not** a classic transit app, timetable viewer, or routing shell.


CatchIt is a **decision-relief and reassurance system for public mobility**.


Its purpose is to compress mobility truth so that people:
- plan less
- doubt less
- check less
- and understand more clearly when they must act and when they do not need to act


---


## 2. Core philosophy
The system must follow this north star:


> A good system does not force people to function better.
> It functions better for people.


This means:
- planning horror is a service problem
- cognitive load reduction is more important than raw data display
- truth matters more than smoothness
- reassurance must never come from concealment or false calm


---


## 3. CatchIt is state-based, not data-first
The design system must not be built as a generic UI kit for transit data.


It must reflect that:
- state logic is more important than process logic
- the same travel truth can be compressed differently across UX layers
- design is not a skin on top of data
- semantic meaning comes before visual styling


---


## 4. Zen terminology
Do not refer to the main calm screen as “the blue screen”.


The correct term is:


**Zen**


Zen is the calm, state-adaptive main expression of the CatchIt core.


Blue is only one state expression within Zen, not the identity of the screen itself.


---


## 5. Cathy is already structurally defined
Do not behave as if Cathy still needs a completely new state system.


The Cathy system states are already defined.


These states can be fully mapped onto Zen.


This means:
- Cathy and Zen are not competing directions
- Cathy and Zen are two expression layers of the same state truth
- Cathy must not be treated as optional decoration
- Cathy must not replace business logic or travel truth


---


## 6. Cathy constraints
Cathy is:
- main UI
- a central USP
- the most important humane translation layer


Cathy is not:
- the travel truth itself
- the business logic
- a free-floating mascot
- a branding toy


Cathy must follow these rules:
- peripheral in stability
- closer in uncertainty
- large only in transition
- never dominant next to critical action content
- should not speak if visual state already communicates enough
- should not emotionalize where precision matters more


---


## 7. Mode separation is mandatory
The product must stay structurally separated into:


- Plan Mode
- Go Mode
- Recovery Mode


These are not stylistic modes.
They are different forms of load reduction.


Do not collapse them into one generic all-purpose screen.


---


## 8. UX layers are part of the product truth
The design system must support multiple compression layers of the same truth, including:


- Orientation
- Comparison
- Transition
- Execution
- Reassurance / Recovery


These are not optional presentation variants.
They are core to the product logic.


---


## 9. Wizard instead of parameter cemetery
The product must not begin with technical preferences, settings complexity, or route-construction UI.


Users should express:
- intentions
- daily context
- movement style
- micro-needs


The system should derive:
- start time
- stop choice
- route
- buffer
- transfers
- return logic
- fallback


The wizard must not feel like profile configuration.


---


## 10. Users express intentions, not routes
The design system must support a product where users express what is happening and what they need, not how they want to manually route themselves.


This includes the chain logic question:


**“Do you need to go anywhere afterwards?”**


This question is a core product differentiator and must be treated seriously.


---


## 11. Active-Move principle is mandatory
Waiting is not neutral.


The system must support the principle that:
- meaningful walking can be better than meaningless waiting
- stability can be better than theoretical speed
- simplicity can be better than aggressive optimization
- real-life movement quality matters, not just ETA


Do not design around ETA alone.


---


## 12. Alternatives must have roles
Alternatives are not “more options”.


They must be semantically differentiated, for example as:
- best choice
- calm reserve
- fallback when disruption happens


Do not present alternatives as flat equivalent cards.


---


## 13. Safety must not become surveillance
CatchIt must not drift into:
- tracking-first logic
- surveillance UX
- safety theater
- alarm inflation
- hidden control patterns


Safety must stay:
- state-based
- trust-based
- escalation-aware
- fallback-ready
- low-stigma
- minimally invasive


---


## 14. No generic design-system clone
Do not output:
- a generic Material clone
- a generic iOS/HIG clone
- a generic SaaS dashboard system
- a generic mobility UI kit


The system must emerge from CatchIt’s semantic structure, not from a premade UI template.


---


## 15. CatchIt2gather is not the active focus
CatchIt2gather is currently only an architectural boundary.


It is:
- defined
- partially prepared
- not the active product focus


The design system must not drift into group UX, social flows, shared live maps, invite/join mechanics, or multi-person coordination as a primary design problem.


Only one active boundary matters:


**The core must remain able to support future multi-person logic later.**


---


## 16. Accessibility is not a late QA layer
Accessibility must be integrated from the beginning.


This includes:
- low interpretation cost
- strong hierarchy
- clear states
- touch target sanity
- color is never the only meaning carrier
- low overload behavior
- support for bad days, stress, ND overload, and low-energy situations


---


## 17. Design system scope for V0.1
The first useful version must stay small and strict.


Do not attempt to generate:
- a complete mature enterprise system
- dozens of polished components
- a final brand language


The first goal is:
- semantic clarity
- strong foundations
- reusable structure
- consistency for non-designers


---


## 18. Final quality check
Any output that violates one of these questions is likely wrong:


- Does this reduce cognitive load, or just look cleaner?
- Does this preserve truth, or just soften it?
- Does this clarify action, or add interpretation?
- Does this fit Plan / Go / Recovery separation?
- Does this keep Zen and Cathy aligned?
- Does this feel like CatchIt, or like a generic transit redesign?
```

# GLOSSARY


## Purpose
This glossary defines the key product, UX, and design-system terms used in CatchIt.


Its purpose is to reduce ambiguity during design-system generation and prevent generic reinterpretation.


If a generated output uses these terms differently, it should be corrected.


---


## CatchIt
A decision-relief and reassurance system for public mobility.


CatchIt is not a classic transit app, route shell, or timetable viewer.
Its purpose is to compress mobility truth so that people plan less, doubt less, check less, and know more clearly when they must act and when they do not need to act.


---


## CatchIt2gather
A future multi-person extension on top of the same CatchIt core.


Not an active product focus.
Not a current design-system focus.
Only relevant as an architectural boundary:
the core must remain able to support future multi-person logic later.


---


## Travel Truth
The underlying mobility reality of the system.


This includes:
- route facts
- timing
- changes
- uncertainty
- confidence
- fallback possibilities
- relevant action implications


Travel Truth is not equal to one screen or one layout.
It can be compressed differently across multiple UX layers.


---


## Decision Relief
A core CatchIt purpose.


Decision Relief means reducing unnecessary cognitive work around mobility:
- fewer unnecessary decisions
- less interpretation burden
- less manual route simulation
- less “just to be safe” checking


CatchIt must reduce decision load, not merely present cleaner data.


---


## Reassurance
A core CatchIt purpose.


Reassurance does not mean aesthetic comfort or false calm.
It means making mobility feel more viable, more understandable, and more trustworthy through:
- state clarity
- action clarity
- confidence legibility
- fallback legibility
- truthful reduction of ambiguity


---


## Planning Horror
The condition CatchIt is built against.


Planning horror is the experience of mobility as:
- mentally heavy
- fragile
- over-interpreted
- uncertainty-dense
- route-simulation-heavy
- constantly re-check dependent


CatchIt treats planning horror as a **service problem**, not a user failure.


---


## Compression
One of the most important terms in CatchIt.


Compression means transforming the same mobility truth into a form that is:
- more usable
- less mentally expensive
- more state-appropriate
- more action-relevant


Compression is not hiding truth.
Compression is shaping truth into a form that supports viable use.


---


## Compression Logic
The product logic that determines how the same truth is expressed differently depending on:
- user state
- urgency
- mode
- context
- level of needed action
- cognitive capacity


This is what makes CatchIt transferable beyond one screen.


---


## State Logic
A way of thinking about the product where meaningful user-facing states matter more than technical process stages.


Examples of meaningful product states:
- stable
- attentive
- fragile
- recheck needed
- fallback ready
- act now
- wait
- observe
- completed


State logic must shape the design system.


---


## Process Logic
A lower-level way of thinking that focuses on technical progression:
- loading
- success
- failure
- route calculated
- provider selected
- update pending


Process logic may exist internally, but must be translated into meaningful product states before reaching UX.


---


## Zen
The calm, state-adaptive main expression of the CatchIt core.


Zen is not “the blue screen”.
Blue is only one state expression within Zen.


Zen is a compression surface that carries the same state truth as other layers, but in a calm, highly reduced, state-adaptive form.


Zen should be understood as:
- a primary state-oriented home expression
- a quiet but semantically strong interface layer
- one of the central UX expressions of CatchIt


---


## Cathy
The main UI and central humane translation layer in CatchIt.


Cathy is:
- a core interface expression
- a central USP
- a humane state-regulation layer


Cathy is not:
- the travel truth itself
- the business logic
- a detached mascot
- mere branding


Cathy system states are already defined and can be fully mapped onto Zen.


---


## Zen / Cathy coupling
The relationship in which Zen and Cathy express the same state truth in different but aligned ways.


Zen expresses:
- system-level state climate
- visual calm or escalation
- semantic atmosphere
- state-adaptive screen behavior


Cathy expresses:
- humane closeness
- regulation
- state intimacy
- transition support


They must not conflict semantically.


---


## Plan Mode
The mode in which CatchIt captures intentions, context, and daily reality.


Plan Mode is not classic route-building.
It is the stage where CatchIt learns:
- what is happening
- when it happens
- whether something follows afterwards
- what movement style is viable
- what micro-needs matter


Plan Mode exists to derive a viable movement style, not to burden the user with manual routing.


---


## Go Mode
The mode in which CatchIt accompanies the live trip.


Go Mode should:
- foreground the next relevant action
- reduce future planning burden
- preserve confidence and fallback legibility
- calm the user by reducing unnecessary trip-wide information


Go Mode is execution-oriented, not planning-oriented.


---


## Recovery Mode
The mode used when something important becomes unstable or breaks.


Recovery Mode must:
- re-establish action clarity
- show the simplest viable rescue path
- avoid panic aesthetics
- avoid full re-planning if possible
- preserve trust even when things change


Recovery is not just an error state.
It is a rescue-of-viability state.


---


## Orientation
A UX layer focused on immediate understanding.


Typical questions:
- What is the current situation?
- Do I need to act now?
- Is the plan stable?
- When do I need to leave?


Orientation is usually very low-noise and high-clarity.


---


## Comparison
A UX layer focused on choosing between a few viable alternatives.


Comparison is not about completeness.
It is about meaningful limited alternatives.


Alternatives must have roles, not just different numbers.


---


## Transition
A UX layer that translates from one mental model into another.


Most importantly:
- from planning to execution
- from stability to increased attention
- from passive confidence to active next-step logic


The step-like structure used during this phase is a **Transition Object**, not a permanent full-screen UI model.


---


## Execution
A UX layer focused on carrying the user through the ongoing trip.


Execution is:
- vertical
- sequential
- reduced
- next-step-focused


Past steps should calm down.
Future steps should remain lightly prepared.


---


## Reassurance Layer
A UX layer focused on stabilizing understanding and reducing anxiety or uncertainty.


It should:
- make trust possible
- reduce unnecessary checking
- preserve truth
- avoid decorative comfort


This layer may overlap with Zen and Cathy behavior.


---


## Recovery / Reassurance distinction
Reassurance can happen without disruption.
Recovery happens when something needs to be actively re-stabilized.


Recovery is sharper.
Reassurance can be gentler.


---


## Wizard
The human-facing input logic for CatchIt.


The wizard must not feel like a settings interface or route-builder.


It should help users express:
- intentions
- daily context
- movement style
- micro-needs


The system should derive mobility structure from that.


---


## Parameter Cemetery
A negative term in CatchIt.


This means any UX that feels like:
- too many settings
- too many toggles
- technical preference management
- configuration burden
- profile management as the primary input experience


The wizard must explicitly avoid becoming this.


---


## Chain Logic
The logic by which CatchIt understands that one plan may continue into another.


A core chain question is:


**“Do you need to go anywhere afterwards?”**


This allows CatchIt to derive:
- linked trips
- timing buffers
- sequential movement realities
- better overall mobility support


This is a distinctive part of CatchIt’s input philosophy.


---


## Active-Move Principle
A core CatchIt principle.


Waiting is not treated as neutral.
Meaningful movement can be better than passive waiting.


This means:
- walking may be preferred over pointless waiting
- route quality is not only ETA
- movement flow matters
- real-life comfort matters
- dead time matters


---


## Viable Movement
A key phrase for CatchIt.


Viable movement means movement that is:
- realistic
- sustainable
- understandable
- stress-aware
- context-aware
- robust enough for actual life


It is not necessarily the mathematically fastest path.


---


## Alternative Roles
Alternatives in CatchIt must carry semantic roles.


Typical roles may include:
- best choice
- calm reserve
- fallback if disruption happens


Alternatives are not flat equivalent route cards.


---


## Action Threshold
The point at which the user must meaningfully act.


The central time expression in Zen should not be a vague countdown.
It should convey a clear action threshold.


Example meaning:
“You must leave within X minutes for the robust plan to hold.”


---


## Confidence
A core product concept.


Confidence refers to how trustworthy, stable, and supported a plan currently is.


Confidence is not decorative.
It is part of truthfulness and reassurance.


---


## Fallback
A viable alternative path or response prepared for when a plan becomes fragile or fails.


Fallback is essential to CatchIt’s trust model.


---


## Recheck
A meaningful product condition where confidence is not yet high enough and the system needs to validate or re-evaluate before claiming too much certainty.


---


## Safety as State Architecture
Safety in CatchIt does not mean:
- surveillance
- tracking-first logic
- alarm inflation
- hidden control mechanisms


Safety means:
- better situational clarity
- clearer escalation boundaries
- earlier fragility detection
- viable fallback preparation
- support without loss of dignity


---


## State Expression
How a product state becomes legible in UX.


State expression may include:
- color
- density
- emphasis
- copy tone
- interaction affordance
- Cathy presence
- Zen intensity
- motion behavior


A state only really exists in UX if it changes expression in a meaningful way.


---


## Semantic Token
A token that represents meaning, not raw visual style.


Examples:
- state.stable
- urgency.high
- confidence.low
- attention.primary
- fallback.available


Semantic tokens should drive visual tokens, not the other way around.


---


## Foundation
The base layer of the design system:
- color
- typography
- spacing
- shape
- motion
- iconography


In CatchIt, foundations must follow semantics, not lead them.


---


## Component
A reusable UI building block.


Examples:
- Zen shell
- primary time block
- alternative card
- transition object
- reassurance banner
- recovery card
- wizard prompt unit


Components must depend on semantic logic, not arbitrary styling alone.


---


## Pattern
A reusable interaction or layout strategy composed of multiple components and rules.


Examples:
- best option + two alternatives
- execution with calming completed steps
- details-on-reveal
- recovery with one rescue direction
- wizard-based movement-style derivation


Patterns are often more important than isolated components in CatchIt.


---


## Bad-Day Resilience
A core success condition.


The system must work not only on ideal, calm, attentive days, but also under:
- stress
- low energy
- overload
- child context
- ND overload
- time pressure
- uncertainty


This is not a niche concern.
It is a core design target.

Which failure modes should be explicitly named in the documentation?


Likely examples:
- generic transit UI drift
- Cathy as branding theater
- Zen reduced to a color theme
- state logic hidden behind pretty surfaces
- wizard becoming parameter cemetery
- recovery becoming panic UI


Need:
Clear anti-pattern section.


---


# Suggested handling
For each open question, choose one of:
- answer now
- defer intentionally
- mark as V0.2 topic
- exclude from scope


Unresolved questions should not silently bleed into the system as vague assumptions.

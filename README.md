# CatchIt Design System Seed


## Purpose
This repository/package is the seed for building a semantic, state-based, reusable design-system architecture for CatchIt.


This is **not** a polished UI project.
It is a structured input package for generating a **V0.1 design-system architecture**.


The goal is to make CatchIt’s product logic operational as a design system:
- calm without hiding truth
- state-based instead of data-first
- reusable beyond single screens
- consistent enough for a non-designer to apply


---


## What CatchIt is
CatchIt is **not** a classic transit app or timetable viewer.


CatchIt is a **decision-relief and reassurance system for public mobility**.


Its purpose is to compress mobility truth so that people:
- plan less
- doubt less
- check less
- understand more clearly when they must act
- and when they do not need to act


It does not only calculate routes.
It supports **viable movement under real-life conditions**.


---


## What this package is for
This package should help generate:


- a design-system north star
- hard design-system principles
- a state taxonomy
- a compression-mode / UX-layer model
- a semantic token architecture
- a V0.1 foundation proposal
- a first component inventory
- reusable patterns
- Zen/Cathy coupling rules
- a documentation structure for Figma and/or code

---


## Important constraints
- Do not start with polished mockups
- Do not invent a new product concept
- Do not drift into generic transit UI
- Do not focus on CatchIt2gather
- Do not treat Zen as just “the blue screen”
- Do not treat Cathy as a mascot detached from semantics
- Start semantic, not visual
- Keep V0.1 small and strict


---


## Directory structure
- `docs/` = source truth and task framing
- `working/` = notes, extracted terms, open questions
- `output/` = generated results
- `archive/` = older prompts or discarded material


---


## Expected first useful output
For the first run, the best target is one single file:


`output/DESIGN_SYSTEM_V0_1.md`


This first document should contain the overall architecture before splitting into:
- tokens
- components
- patterns
- implementation structure


---


## Success criteria
A successful output should:
- feel like a real design-system architecture
- preserve CatchIt’s semantic core
- help a non-designer build consistently
- be reusable across multiple UX surfaces
- keep Zen and Cathy aligned as expression layers of the same state truth


The output fails if it:
- becomes branding-first
- becomes generic transit UI
- ignores state logic
- ignores mode separation
- collapses into a random UI kit

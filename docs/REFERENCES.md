# REFERENCES


## Purpose
This document collects structural and directional references for the CatchIt design-system work.


These references are not here to be copied.
They are here to clarify what kind of system-thinking, documentation, and design discipline feels useful.


---


## 1. Marvel Styleguide
Reference:
https://marvelapp.com/styleguide/


Why it is relevant:
- It suggests a useful way of thinking in systems instead of isolated screens
- It appears to separate foundations, components, and patterns in a clear way
- It feels like a helpful structural reference for how a design system can be documented
- It is relevant more as a documentation and organization inspiration than as a visual target


Important note:
Do not copy Marvel’s visual language.
Use it only as an example of structured system presentation.


---


## 2. W3C / WCAG accessibility principles
Reference:
https://www.w3.org/WAI/standards-guidelines/wcag/


Why it is relevant:
- Accessibility must be part of the system from the beginning
- CatchIt needs clarity under stress, low energy, overload, and low interpretation capacity
- The principles of perceivable, operable, understandable, and robust are highly relevant
- This is especially important because CatchIt must work for bad days, not just ideal usage


Important note:
Accessibility is not only a compliance layer.
For CatchIt, accessibility is part of the product philosophy.

## 3. CatchIt’s own semantic core
Primary internal reference:
- CATCHIT SSOT
- CATCHIT2GATHER SSOT


Why it is relevant:
- These are the primary truth sources
- External references must not override the product’s own semantic structure
- The design system must emerge from CatchIt’s own states, layers, and mode logic


Important note:
If an external reference contradicts CatchIt’s product truth, CatchIt wins.


---


## 4. What to look for in references generally
Useful references should help with:
- structural rigor
- documentation clarity
- foundation logic
- reusable patterns
- semantic consistency
- accessibility
- low-ambiguity design rules


Less useful references are those that mostly contribute:
- aesthetic trends
- dribbblified polish
- transit clichés
- visual novelty without semantic depth


---


## 5. What not to imitate
Do not imitate:
- generic transit apps whose main surface is raw route data
- SaaS admin systems with dashboard logic
- design systems that prioritize brand flair over semantic rigor
- systems that treat characters or mascots as branding instead of regulated UX layers
- systems that hide weak logic behind smooth visuals


---


## 6. What to extract from references
Whenever a reference is used, extract only:
- structural patterns
- documentation ideas
- semantic clarity methods
- accessibility discipline
- good hierarchy behavior
- reusable token logic


Do not extract:
- surface style blindly
- fashionable color choices
- arbitrary typography character
- decorative motion language
- generic card spam


---


## 7. Preferred attitude toward references
References should be treated as:
- scaffolding
- comparison material
- organizational inspiration
- prompts for system discipline


They should not be treated as:
- final answers
- identity substitutes
- ready-made aesthetics
- shortcuts around product thinking


---


## 8. Reference discipline rule
For every external reference, ask:
- What is useful here structurally?
- What is dangerous here semantically?
- Does this support CatchIt’s core, or dilute it?
- Is this helping with consistency, or just making things look “more designed”?


If the answer is mostly “it looks polished”, it is probably not a good reference for this phase.

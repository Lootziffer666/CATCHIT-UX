# Drift Prevention

## Common wrong directions
- Converting CatchIt into generic transit search/list behavior.
- Moving key decisions into settings-heavy backstage flows.
- Treating map-first layouts as primary orientation.
- Reframing the product as dashboard/status monitoring.

## What drift looks like
- **Generic transit-app drift**: route results dominate, reassurance logic weakens.
- **Settings drift**: day-critical choices hidden in durable preferences.
- **Dashboard drift**: operational flow replaced by passive metrics overview.
- **Map-first drift**: map becomes default truth instead of state/anchor truth.
- **Archive confusion**: prototype artifacts copied forward as if canonical.

## Practical anti-regression checks
- Does Main remain current truth?
- Are Side Swipe and Backstage still distinct?
- Is planning horizontal and execution vertical?
- Are Core Gold / Extended Gold boundaries intact?
- Are prototype insights explicitly promoted before becoming normative?

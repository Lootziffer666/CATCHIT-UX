# CONSTRAINTS


## Purpose
This document defines the practical and conceptual constraints for the CatchIt design-system work.


The point is not to limit quality.
The point is to prevent drift, bloat, and false starts.


---


## 1. The creator is not a trained visual designer
This system must be buildable and usable by someone with strong product intuition but limited formal design training.


Therefore the system must be:
- explicit
- low-ambiguity
- systematic
- forgiving
- easy to apply consistently


Avoid outputs that assume elite visual craft from day one.


---


## 2. The first version must be small
This is a V0.1 design system.


It should begin with:
- semantic structure
- a practical foundation layer
- a small component inventory
- reusable patterns
- clear rules


It must not begin with:
- massive libraries
- dozens of component variants
- final brand language
- extreme visual sophistication


---


## 3. Start semantic, not visual
The system must be built in this order:


1. product philosophy
2. principles
3. states
4. compression logic
5. semantic tokens
6. foundations
7. components
8. patterns


Do not start with:
- font shopping
- color styling
- UI polish
- moodboards


---


## 4. Zen and Cathy are already grounded in product logic
Do not treat Zen and Cathy as a blank-slate stylistic playground.


Assume:
- Zen is a state-adaptive main expression of the core
- Cathy is a defined humane UI layer
- Cathy states already exist
- Zen can inherit and express the same system states


The task is to structure and operationalize this, not invent a disconnected aesthetic concept.


---


## 5. CatchIt2gather is out of scope
CatchIt2gather is not the active product focus.


The design system should only account for it as a future architectural boundary.


Do not spend design energy on:
- group screens
- social mechanics
- shared maps
- invites
- join flows
- participant roles
- group orchestration


---


## 6. Accessibility must be assumed from the beginning
The system should assume:
- bad days
- low energy
- stress
- time pressure
- ND overload
- low interpretation bandwidth


Accessibility must not be reduced to:
- contrast only
- compliance checklists only
- “we’ll fix that later”


The system must minimize interpretation cost.


---


## 7. No dependence on final brand assets
Assume:
- no final font chosen yet
- no polished brand palette yet
- no final icon set yet
- no refined motion language yet


The system must propose workable interim foundations for V0.1.


---


## 8. Consistency matters more than beauty in V0.1
The system should optimize for:
- consistency
- clarity
- state readability
- cognitive calm
- reusability


Do not optimize primarily for:
- expressiveness
- novelty for its own sake
- ornamental richness
- impressive visual polish


---


## 9. Avoid generic design-system defaults
Do not assume the right answer is:
- Material Design
- Human Interface Guidelines
- enterprise dashboard patterns
- default transit UI patterns


Generic systems may be useful references, but CatchIt’s system must come from its product truth.


---


## 10. No data-first output
This system must not encourage:
- timetable UI as the default
- data dumps
- route-first thinking
- provider logic in the surface
- raw technical status exposure


All outputs must prefer:
- next meaningful action
- time threshold clarity
- state clarity
- trust and fallback legibility
- role-based alternatives


---


## 11. Motion must stay calm and meaningful
Motion should support:
- transition
- state shift
- focus
- reassurance
- escalation clarity


Motion should not become:
- decorative animation
- attention bait
- personality theater
- unnecessary movement load


---


## 12. The result must help real building
The output should be usable for:
- Figma setup
- token naming
- documentation
- implementation planning
- future code structure


It should not remain abstract philosophy only.


---


## 13. Every recommendation must justify itself
Any proposed rule, token, component, or pattern should explain:


- why it exists
- what CatchIt problem it solves
- what load it removes
- what bad design tendency it prevents


If it cannot justify itself, it probably does not belong in V0.1.


---


## 14. Use practical defaults where needed
If a final choice is unknown, recommend pragmatic temporary defaults.


Examples:
- a working type setup instead of final brand typography
- a semantic color system instead of final brand palette
- a strict spacing scale instead of “designer intuition”
- a small radius scale instead of ad hoc corner rounding


---


## 15. Help a non-designer stay consistent
This is one of the most important constraints.


The system must be strong enough that someone with:
- clear product thinking
- good intuition
- limited design craft


can still create work that feels coherent.


That is one of the main success conditions.
